Start with the uncertainty—not the ceremony.

Select the statement closest to your task. The mode changes the route, not the quality bar.

Survey

Clarify the few product, design, architecture, or quality-bar unknowns that could change the implementation route.

Use whenTwo or more meaningful directions remain viable.
Exit withA supported target and an explicit definition of success.

Track

Follow evidence from symptom to owning seam. A failing test, log, or report begins the investigation; it does not get to dictate the fix.

Use whenBehavior is failing, flaky, slow, regressed, or inconsistent.
Exit withA supported root cause and the smallest responsible repair.

Prove

Define one expected behavior with a focused failing test, then make the smallest change that turns that evidence green.

Use whenThe behavior is already understood and tightly bounded.
Exit withA passing proof plus owned-diff review and validation.

Expedition

Plan, implement, validate, and review a scoped change when the target and owning seam are clear enough to move.

Use whenThe main uncertainty is execution rather than intent or cause.
Exit withA verified result, explicit risks, and authorized delivery state.

Not every unknown needs a meeting.

The taxonomy is a discovery checklist. It tells the agent whether to inspect, ask, prototype, or simply proceed.

KNOWN / KNOWN

Constraints

Honor the explicit request, trusted instructions, and verified repository conventions.

KNOWN / UNKNOWN

Questions

Resolve only decisions that can materially change architecture, behavior, security, UX, or validation.

UNKNOWN / KNOWN

Recognition

Use a prototype or example when taste, density, or interaction is easier to recognize than describe.

UNKNOWN / UNKNOWN

Blind spots

Inspect coupling, ownership, compatibility, accessibility, deployment, and evidence quality.

One route. Observable checkpoints.

Discovery stops when the owning seam, relevant contract, worktree state, and validation route are supported.

01 · Orient

Enter the real repository

Capture the branch, worktree baseline, trusted context, and task-owned files before changing anything.

02 · Route

Decide what could change the plan

Classify the important unknowns and choose the cheapest technique that can close them.

03 · Change

Patch the smallest valuable surface

Preserve unrelated work, avoid silent scope expansion, and record only material deviations.

04 · Verify

Classify validation honestly

Separate task-caused failures, pre-existing failures, and environmental blockers.

05 · Review

Try to disconfirm the result

Review the owned delta from the original request and evidence, not from the implementation's intent.

06 · Report

State the actual territory

Hand off what changed, what passed, what remains uncertain, and exactly how far delivery was authorized.

Evidence has boundaries.

The repository distinguishes deterministic integrity from behavioral claims instead of letting one stand in for the other.

Deterministic checks

Package structure, aligned manifests, evaluation records, artifact contracts, and unit tests run without model calls.

Existing behavior evidence

Qualified synthetic and real-repository reports are preserved with retry history and disclosed limitations.

Current limitation

The current progressive-disclosure structure has not received a fresh attempt-26 blinded cross-family behavioral cohort. The lock preregisters future evaluation; it does not claim that evaluation already happened.
